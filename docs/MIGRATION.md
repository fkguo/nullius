# Migrating an External Project from `autoresearch` (≤ 0.3.0) to `nullius` (≥ 0.4.0)

In 0.4.0 the project was renamed `autoresearch-lab` → `nullius` in one mechanical
change with **no compatibility layer**: the CLI, the project state directory, the
sentinel formats, the `@nullius/*` package scope, and the `NULLIUS_*` environment
prefix all switched at once. This guide covers external research projects whose
root contains a `.autoresearch/` state directory created by 0.3.0 or earlier.

## You do not have to migrate

`v0.3.0` is a tagged, pushed resting point. A project that stays there keeps its
`.autoresearch/` state untouched — the 0.4.0+ CLI does not read, migrate, or
delete it. If you need to *operate* such a project with the old CLI, build it
from the tag in a separate checkout:

```bash
git -C /path/to/nullius worktree add ../autoresearch-v0.3.0 v0.3.0
cd ../autoresearch-v0.3.0 && pnpm install && pnpm -r build
node packages/orchestrator/dist/cli.js status --project-root /path/to/project --json
```

## Migration steps (per project)

Prerequisite: 0.4.0 is built and the `nullius` command is on your `PATH`
(or substitute `node /path/to/nullius/packages/orchestrator/dist/cli.js`).

```bash
cd /path/to/your-project

# 1. Rename the state directory. Lifecycle data moves as-is.
mv .autoresearch .nullius

# 2. Drop the old launcher; it will be regenerated under the new name.
rm .nullius/bin/autoresearch

# 3. Rebuild the runtime layer: new launcher + new HARNESS sentinel.
nullius init --runtime-only --project-root "$PWD"

# 4. Verify continuity: tasks, approvals, and the ledger must all still be there.
./.nullius/bin/nullius status --json

# 5. Refresh managed scaffold docs (templates that mention the old name).
nullius init --refresh --project-root "$PWD"
```

What each layer does during migration:

| Surface | What happens |
|---|---|
| `state.json`, `ledger.jsonl` | Carried over **byte-identical** — verified by sha256 in the 0.4.0 release rehearsal. Neither file embeds the old name. |
| `HARNESS` sentinel | Rewritten unconditionally by step 3 (`kind: nullius_project_harness`). Until then the new CLI reports `NULLIUS_HARNESS_SENTINEL_INVALID_CONTRACT` and points you at step 3. |
| `HARNESS_INVOCATION` receipt | Rewritten automatically on every `status` call; nothing to do. |
| `.nullius/bin/nullius` launcher | Regenerated by step 3 with the new name and current dist path. |
| Scaffold docs (`AGENTS.md`, `project_index.md`, …) | Step 5 updates managed files (backing up any you modified to `.nullius/backups/`) and leaves user-owned files untouched; it prints a per-file list. |

## Related surfaces outside the project root

- **Environment variables**: anything you exported as `AUTORESEARCH_*` (shell
  profiles, CI, MCP server `env` blocks) must become `NULLIUS_*`. There is no
  fallback probing of old names.
- **MCP configs with absolute paths**: entries pointing into the development
  checkout (e.g. `/…/autoresearch-lab/packages/hep-mcp/dist/index.js`) keep
  working until that directory itself is moved or renamed; update them together
  with that move.
- **Global scratch data**: the HEP scratch fallback moved `~/.autoresearch/` →
  `~/.nullius/`. Project-rooted data (`<project>/artifacts/hep-mcp`) is
  unaffected. The old global cache can be moved (`mv ~/.autoresearch ~/.nullius`)
  or simply abandoned — it holds re-downloadable material only.
